# AIPlayer 架构设计文档

## 1. 项目目标

AIPlayer 是一个面向桌面端的跨平台 AI 视频播放器，目标是在本地或半本地环境中实现以下能力：

- 播放本地视频、音频与常见流媒体资源。
- 在播放过程中提取音频流，送入语音识别模型进行转写。
- 将识别结果进一步翻译为目标语言，生成双语字幕。
- 通过 mpv 的 IPC / API 动态挂载、刷新和渲染字幕，实现“边播边出字幕”的体验。
- 在 macOS、Windows、Linux 上尽量保持一致的 UI 与核心能力。

该项目的核心价值不只是“播放器 + 字幕”，而是构建一条可扩展的多阶段 AI 媒体处理流水线：**媒体解复用 -> 音频切片 -> ASR -> 分段对齐 -> 翻译 -> 字幕合成 -> 播放器动态渲染**。

## 2. 技术选型

### 2.1 跨平台 GUI：PySide6 / PyQt6

推荐优先级：**PySide6 > PyQt6**。

原因如下：

1. **跨平台成熟**：Qt 在桌面 GUI 领域非常成熟，适合构建播放器、设置面板、字幕控制面板、任务队列窗口等复杂桌面应用。
2. **Python 生态友好**：PySide6 / PyQt6 都有成熟绑定，便于快速开发原型。
3. **多媒体与窗口嵌入能力强**：虽然实际播放建议交给 mpv，但 Qt 非常适合承载播放器窗口、控制条、字幕样式设置、任务状态面板。
4. **打包分发经验丰富**：后续可结合 PyInstaller / Nuitka 做桌面分发。
5. **许可证角度**：PySide6 基于 LGPL，商业与开源场景通常更灵活，因此更推荐。

建议：

- **MVP 阶段**：使用 `PySide6`。
- 若团队已有 PyQt6 经验，也可替换，但应尽量避免混用。

### 2.2 播放器内核：python-mpv + libmpv

推荐组合：**python-mpv（Python 封装） + libmpv（底层播放器内核）**。

原因如下：

1. **mpv 播放能力强**：支持大量封装格式、字幕、滤镜、硬件解码、网络流媒体。
2. **libmpv 可嵌入**：可以嵌入 Qt 窗口中，而不是单独弹出播放器进程。
3. **字幕控制能力强**：支持动态加载字幕轨、切换字幕、ASS 样式渲染。
4. **IPC / JSON API 易于联动**：适合在 AI 流水线产出新字幕片段后，动态通知播放器刷新。
5. **社区成熟**：大量项目已验证 mpv 适合作为“可编程播放器内核”。

建议实现方式：

- Python 层通过 `python-mpv` 控制播放、暂停、seek、加载字幕。
- 对于更复杂的控制，可同时启用 mpv 的 `input-ipc-server`，通过 JSON IPC 做补充控制。
- 字幕渲染优先使用：
  - **SRT**：简单、兼容性高。
  - **ASS**：用于双语排版、字体、颜色、阴影、上下双行布局。

### 2.3 AI 语音识别：Faster-Whisper

推荐：**faster-whisper**。

原因如下：

1. 相比原始 Whisper，推理速度更快，资源占用更可控。
2. 支持 CPU / GPU 场景，适合桌面端部署。
3. 支持分段时间戳输出，便于字幕对齐。
4. 社区使用广泛，适合做 MVP 与后续优化。

建议策略：

- **MVP**：离线整段识别，先生成完整字幕文件。
- **V2**：按音频块流式识别，维护增量字幕缓存。
- 模型选择：
  - 低配设备：`small` / `medium`
  - 高配设备：`large-v3` 或后续兼容模型

### 2.4 翻译能力：MarianMT / 外部大模型 API

翻译层建议采用“双通道策略”：

#### 方案 A：MarianMT（本地翻译）

优点：

- 可离线运行。
- 成本可控。
- 适合隐私敏感场景。

缺点：

- 翻译质量通常不如新一代大模型。
- 对口语、上下文、省略句的处理较弱。

适用：

- MVP / 离线模式 / 无网络模式。

#### 方案 B：外部大模型 API

可接入 OpenAI 兼容接口、Claude 类接口或其他翻译 / LLM 服务。

优点：

- 翻译质量高。
- 可做术语统一、上下文润色、双语对齐优化。
- 可扩展为摘要、单词解释、台词讲解等高级功能。

缺点：

- 有网络依赖与调用成本。
- 需要处理速率限制、失败重试、隐私合规。

适用：

- V2 / V3 高级模式。

#### 推荐落地方式

定义统一翻译接口：

- `Translator` 抽象基类
- `MarianTranslator`
- `LLMTranslator`

这样可以在 UI 中切换：

- 本地翻译
- 在线高质量翻译
- 原文字幕模式

### 2.5 音频处理与媒体辅助

建议依赖：

- `ffmpeg-python`：用于音频抽取、转码、切片。
- `numpy`：音频缓冲、特征处理、数据拼接。
- 必要时可引入 `soundfile` / `pydub` / `webrtcvad`：
  - `soundfile`：读写音频
  - `pydub`：简单切片
  - `webrtcvad`：语音活动检测，减少无效识别

### 2.6 并发与任务调度

建议：

- GUI 主线程：Qt 事件循环
- 播放控制线程：mpv 事件监听
- AI 处理线程池：ASR / 翻译 / 字幕合成
- 队列机制：`queue.Queue` 或 `asyncio` + 线程桥接

推荐原则：

- **UI 与 AI 推理绝对隔离**，避免界面卡死。
- **播放器时钟作为主时钟源**，字幕只做跟随，不反向驱动播放。

## 4. 核心流水线设计（Data Flow）

这一部分是整个项目的关键。

目标：**边播放视频，边实时提取音频流，送入 AI 模型识别与翻译，并通过 mpv 动态渲染双语字幕。**

### 4.1 主流程概览

```text
用户打开视频
  -> mpv 开始播放
  -> 后台音频提取器从媒体源持续抽取音频
  -> 音频切片器按时间窗切分（如 2s / 5s）
  -> VAD / 缓冲器过滤静音与拼接上下文
  -> ASR 引擎输出带时间戳的转写片段
  -> 翻译引擎生成目标语言文本
  -> 字幕合成器生成 SubtitleCue
  -> 字幕缓存区写入 ASS / SRT 增量文件
  -> mpv 通过 IPC / API 重新加载或刷新字幕轨
  -> 用户看到动态更新的双语字幕
```

### 4.2 音频提取策略

有两种主流方案：

#### 方案 1：从原始媒体文件并行抽取音频

做法：

- 播放器播放视频。
- 后台使用 ffmpeg 从同一媒体源读取音频流。
- 将音频转为统一格式，例如：
  - 单声道
  - 16kHz
  - PCM float / int16

优点：

- 不依赖播放器回传音频数据。
- 实现简单，稳定性高。
- 适合本地文件与大多数网络资源。

缺点：

- 需要自己处理播放时间与识别时间的同步。

#### 方案 2：从 mpv 音频输出链路截取

做法：

- 通过 mpv 的音频滤镜、hook 或外部音频输出链路获取播放中的音频数据。

优点：

- 与实际播放音频完全一致。

缺点：

- 实现复杂。
- Python 侧接入成本高。
- 跨平台稳定性不如方案 1。

**推荐：先落地方案 1。**

### 4.3 时间同步设计

实时字幕最难的不是“识别”，而是“对齐”。

建议采用以下同步原则：

1. **播放器时间为真值**：以 mpv 当前 `playback-time` 为主时钟。
2. **音频块带源时间戳**：每个 `AudioChunk` 必须记录：
   - `source_start`
   - `source_end`
3. **ASR 输出相对时间 -> 绝对时间**：
   - 若 ASR 返回 chunk 内相对时间戳，则加上 `source_start` 转为媒体绝对时间。
4. **翻译不改变时间轴**：翻译只替换文本，不改 cue 的起止时间。
5. **增量修正机制**：流式识别前几秒可能不稳定，允许对最近 N 秒字幕进行覆盖更新。

### 4.4 分块与缓冲策略

建议参数：

- 音频切片窗口：2~5 秒
- 上下文重叠：0.5~1.5 秒
- 字幕提交延迟：0.5~2 秒

原因：

- 切片太短：识别上下文不足，错误率高。
- 切片太长：实时性差。
- 适度重叠可减少句子被截断的问题。

推荐机制：

- `ChunkBuffer`：维护最近若干秒音频。
- `SegmentStabilizer`：对 ASR 的临时结果与最终结果做合并。
- `SubtitleWindow`：只保留当前播放点附近的字幕片段用于动态刷新。

### 4.5 ASR 识别流程

```text
AudioChunk
  -> 预处理（重采样 / 降噪 / VAD）
  -> Faster-Whisper 推理
  -> 输出 TranscriptSegment[]
  -> 绝对时间对齐
  -> 置信度过滤
  -> 送入翻译队列
```

建议输出结构：

- `id`
- `start_time`
- `end_time`
- `source_text`
- `confidence`
- `is_final`

### 4.6 翻译流程

```text
TranscriptSegment
  -> 文本清洗（去口头禅、修正断句、合并碎片）
  -> Translator.translate()
  -> 输出 TranslationSegment
  -> 与原文合并为双语字幕片段
```

建议翻译前增加一层“文本整形”：

- 合并过短片段
- 修复标点
- 统一专有名词
- 对连续片段做上下文翻译

这样能显著提升双语字幕可读性。

### 4.7 字幕合成与渲染

#### SRT 模式

适合：

- MVP
- 导出字幕文件
- 兼容性优先

格式简单，但双语排版能力有限。

#### ASS 模式

适合：

- 实时双语字幕
- 自定义字体、颜色、描边、阴影
- 上下双行布局

推荐双语排版：

- 第一行：原文（白色）
- 第二行：译文（青色 / 黄色）
- 使用较强描边，保证视频背景下可读性

### 4.8 mpv 动态渲染方案

可选方案如下：

#### 方案 A：持续写入字幕文件并 reload

做法：

- 后台维护一个临时 `.ass` 或 `.srt` 文件。
- 每次新增字幕后，通知 mpv 重新加载字幕轨。

优点：

- 实现简单。
- 易于调试。

缺点：

- 高频 reload 可能闪烁或有性能损耗。

#### 方案 B：维护外部字幕轨并低频刷新

做法：

- 每隔 1~2 秒批量刷新一次字幕文件。
- 只更新最近窗口内的字幕。

优点：

- 性能更稳。
- 用户体验更平滑。

#### 方案 C：通过脚本 / IPC 做更细粒度控制

做法：

- 借助 mpv 脚本层（Lua / JS）或更底层 API，动态注入字幕内容。

优点：

- 灵活度最高。

缺点：

- 开发复杂度高。

**推荐路线：MVP 用 A，V2 升级到 B，V3 再评估 C。**

### 4.9 错误恢复与容错

必须考虑以下异常：

- ffmpeg 音频抽取中断
- ASR 模型加载失败
- 翻译 API 超时 / 限流
- 字幕文件写入失败
- mpv reload 字幕失败

建议机制：

- 任务状态机：`idle / running / degraded / failed`
- 重试策略：
  - ffmpeg：短重试
  - LLM API：指数退避
- 降级策略：
  - 翻译失败时仅显示原文
  - 实时失败时回退到离线识别模式

## 7. 分阶段实现路径

### 7.1 MVP 阶段：离线识别后挂载字幕

目标：**先做成、先可用、先闭环。**

功能范围：

1. 打开本地视频文件。
2. 使用 ffmpeg 抽取完整音频。
3. 使用 Faster-Whisper 离线识别整段音频。
4. 使用 MarianMT 或可选在线翻译生成译文。
5. 生成 `.srt` / `.ass` 双语字幕文件。
6. 通过 mpv 挂载字幕播放。
7. 提供基础 GUI：打开文件、开始识别、加载字幕、切换字幕显示。

MVP 成功标准：

- 用户能在桌面端打开视频。
- 能一键生成双语字幕。
- 能在播放器中正常显示字幕。
- 整体流程稳定，不追求实时。

为什么先做 MVP：

- 风险最低。
- 最容易验证“播放器 + AI 字幕”产品价值。
- 可先打通模型、字幕、播放器三者之间的接口。

### 7.2 V2 阶段：实时流式识别与翻译

目标：**从“离线工具”升级为“实时 AI 播放器”。**

新增能力：

1. 播放时并行抽取音频流。
2. 按时间窗切片并送入 ASR。
3. 增量生成识别结果。
4. 增量翻译并合成双语字幕。
5. 动态刷新 mpv 字幕轨。
6. 支持字幕修正与稳定化。

关键难点：

- 时间同步
- 增量字幕覆盖
- 模型推理延迟控制

V2 成功标准：

- 播放过程中可持续出现字幕。
- 延迟控制在可接受范围（例如 2~5 秒）。
- seek / pause / resume 后系统能恢复同步。

## 8. 关键工程决策建议

### 8.1 先离线、后实时

不要一上来就做全实时。先把：

- 音频抽取
- ASR
- 翻译
- 字幕生成
- mpv 挂载

这条链路打通，再做流式化。

### 8.2 先 SRT、后 ASS 强化

- MVP 先保证 SRT 正常。
- V2 / V3 再重点优化 ASS 双语样式。

### 8.3 统一抽象接口，避免后期重构爆炸

建议一开始就抽象：

- `PlayerBackend`
- `ASREngine`
- `Translator`
- `SubtitleRenderer`

这样后续替换模型、替换翻译服务、替换字幕策略时成本更低。

### 8.4 缓存与中间产物必须可追踪

建议缓存：

- 抽取音频文件
- ASR 原始 JSON
- 翻译结果 JSON
- 最终 SRT / ASS

好处：

- 便于调试
- 便于断点恢复
- 便于后续做“重新翻译但不重跑识别”

## 9. 性能与资源考量

### 9.1 CPU / GPU 策略

- 无 GPU：优先 `small` / `medium` 模型
- 有 GPU：可启用更大模型
- UI 中应明确提示模型资源消耗

### 9.2 内存控制

- 长视频不要一次性全部载入内存
- 流式模式下只保留最近窗口数据
- 大模型实例尽量复用，避免频繁初始化

### 9.3 延迟优化

可从以下方向优化：

- 缩短音频块长度
- 使用 VAD 减少无效片段
- 翻译批处理
- 对最近字幕窗口做低频批量刷新
- 使用更快模型或量化模型

---

## 10. 安全、隐私与合规

如果接入外部大模型 API，需要明确：

- 用户音频 / 文本是否上传云端
- 是否保留日志
- API Key 如何本地安全存储
- 是否允许完全离线模式

建议：

- 默认优先本地识别
- 在线翻译需显式开关
- 在设置中清晰提示数据流向

---

## 11. 最小可执行落地建议

如果现在立刻开始编码，建议第一周只做以下内容：

1. 用 PySide6 做一个主窗口。
2. 用 python-mpv 嵌入播放器并能打开本地视频。
3. 用 ffmpeg 抽取音频。
4. 用 faster-whisper 生成原文 SRT。
5. 再接一个翻译器生成双语 SRT / ASS。
6. 在 mpv 中加载字幕并验证显示效果。

只要这 6 步跑通，项目就已经不是 PPT，而是一个真正可迭代的 AI 播放器雏形。

---

## 12. 结论

这个跨平台 AI 播放器方案的最佳落地路径，不是一步到位做“全实时智能播放器”，而是：

1. **先用 PySide6 + python-mpv + faster-whisper 打通离线闭环**；
2. **再把离线识别升级为流式识别与增量字幕刷新**；
3. **最后补齐 UI、性能优化、打包分发与高级 AI 能力**。

从工程可行性、开发效率、跨平台稳定性、后续扩展性来看，这是一条风险最低、收益最高的路线。

如果按这个架构推进，AIPlayer 完全可以从一个“带 AI 字幕的播放器”，逐步演进成一个“面向语言学习、内容理解、跨语种观看”的桌面 AI 媒体平台。
